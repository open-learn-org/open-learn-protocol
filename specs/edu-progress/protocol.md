# EduProgress v1 — Progress Reporting for Learning Apps

*Version 1 · open draft · feedback welcome*

If you build a learning app, you already know what the child is doing inside it. You know when they started, what they tried, what they got right, what they got stuck on. That data is interesting to exactly one party who can't see it right now: the parent or teacher who set the child up in the first place.

EduProgress is the smallest spec we could write that lets you hand that data to them, in a shape they can use, without writing a custom integration per pair of (your app, their dashboard).

It's a sibling to [EduSSO](../edu-sso/): EduSSO tells you who the child is, EduProgress tells someone else what they did.

---

## The idea in one paragraph

Your app emits structured events to an HTTPS endpoint we call the *collector*. The events follow a JSON schema with a small mandatory envelope (event id, timestamp, child reference, app reference, event type) plus a type-specific payload. You batch them, you POST them, the collector ACKs. There is no streaming, no shared skill taxonomy, no analytics — only ingest. The collector decides what to do with the events.

The protocol is intentionally narrow because every previous attempt at a shared learning-data spec (xAPI, Caliper) drowned in trying to model pedagogy. We aren't going to model pedagogy. We're going to define an envelope and let you put what you know inside it.

## The flow

```
┌────────────┐  1. authenticated POST   ┌──────────────┐
│ Learning   │ ──────────────────────►  │ Collector    │
│ app        │   { events: [...] }      │ (launcher,   │
│ (reporter) │                          │  dashboard)  │
│            │ ◄──────────────────────  │              │
└────────────┘   202 { accepted: N }    └──────────────┘
```

1. At onboarding the collector tells the reporter: an ingest URL, a bearer token, and (optionally) the identity model to use for `child_ref`.
2. The reporter accumulates events as they happen. When the buffer hits a threshold or a timer fires, it POSTs them in one batch.
3. The collector validates each event against the schemas, accepts what's valid, rejects what isn't, and returns a per-event status.

No long-poll, no webhooks back, no streaming. The reporter does not learn anything from the collector except "I accepted N of M events."

## The envelope

Every event has the same outer shape:

```json
{
  "event_id": "01HX5XYV6FPK3R3D6T2H8E2VPR",
  "type": "activity.completed",
  "occurred_at": "2026-05-20T14:32:17.412Z",
  "received_at": null,
  "child_ref": {
    "iss": "https://issuer.example-launcher.com",
    "sub": "child:abc123",
    "email": "student@example.com"
  },
  "app_ref": {
    "app_id": "your-app-id",
    "version": "2.4.1"
  },
  "session_id": "01HX5XYTX5N3DTSXJC9MFAV0QH",
  "data": { /* type-specific payload */ }
}
```

- `event_id` — ULID or UUIDv7 generated by the reporter. Used by the collector for idempotency.
- `type` — one of the registered event types, in dotted notation. See "Core event types" below.
- `occurred_at` — RFC 3339 timestamp, UTC, when the event happened in the reporter's clock.
- `received_at` — set by the collector when persisting; reporters leave it `null`.
- `child_ref` — at least one of `(iss, sub)`, `email`, or a collector-assigned `child_id`. Use whichever the collector accepts. If EduSSO is in play, `(iss, sub)` is the right answer.
- `app_ref` — the app's stable id with the collector, plus an optional version string for debugging.
- `session_id` — opaque correlation id grouping events from one logical session. Same id across `session.started` → ... → `session.ended`.
- `data` — type-specific. Validated against the schema for `type`.

The envelope schema is in [`schema/event.schema.json`](./schema/event.schema.json).

## Core event types

v1 defines six. None of them are optional in the sense that you must emit all six — emit what you observe. But you may not invent new top-level types in v1; if you need one, propose it as an RFC.

### `session.started`

Emit when a child begins using your app.

```json
{
  "type": "session.started",
  "data": {
    "started_at": "2026-05-20T14:00:00Z",
    "client": {
      "platform": "desktop",
      "launcher": "https://issuer.example-launcher.com"
    }
  }
}
```

### `session.ended`

Emit when the session ends (child closes, timeout, parent intervention).

```json
{
  "type": "session.ended",
  "data": {
    "ended_at": "2026-05-20T14:32:00Z",
    "active_ms": 1680000,
    "reason": "closed"
  }
}
```

`reason` is one of `closed | timeout | switched | unknown`.

### `activity.started`

Emit when the child begins an identifiable unit of work — a lesson, an exercise, a level, whatever your app's atom of activity is.

```json
{
  "type": "activity.started",
  "data": {
    "activity_id": "fractions-intro-3",
    "topic": "math.fractions",
    "difficulty": 2
  }
}
```

`topic` is an opaque dotted string. We do *not* maintain a registry. If two apps both emit `math.fractions`, that's because they both decided to; it's not enforced.

### `activity.completed`

Emit when an activity ends with a result.

```json
{
  "type": "activity.completed",
  "data": {
    "activity_id": "fractions-intro-3",
    "score": 0.85,
    "time_ms": 124000,
    "attempts": 3,
    "passed": true
  }
}
```

`score` is `0.0` to `1.0`. `passed` is your app's notion of "good enough." If your app doesn't grade, omit both.

### `skill.progress`

Emit when the child's mastery of a skill changes meaningfully. Don't emit one per micro-event; emit when the change is real.

```json
{
  "type": "skill.progress",
  "data": {
    "skill_id": "math.fractions.adding",
    "level": 0.62,
    "delta": 0.08,
    "evidence": ["activity:fractions-intro-3"]
  }
}
```

`level` is `0.0` to `1.0`. `delta` is the change since the last reported value; the collector does not have to compute it. `evidence` is an optional list of opaque references the collector can show to a parent ("we learned this from these three activities").

### `struggle.signal`

Emit when you detect that the child is stuck, confused, or thrashing. This is the event that most differentiates EduProgress from earlier learning-data specs: we want negative signals, not just achievements.

```json
{
  "type": "struggle.signal",
  "data": {
    "kind": "stuck",
    "context": {
      "activity_id": "fractions-intro-3",
      "duration_ms": 45000
    }
  }
}
```

`kind` is one of `stuck | rapid_click | repeated_error | abandoned`. The `context` shape varies; we constrain it loosely in the schema to allow reasonable evolution.

## Transport

### Endpoint

```
POST https://collector.example.com/v1/events
Authorization: Bearer <reporter_token>
Content-Type: application/json
Idempotency-Key: <ulid>          // optional but recommended

{
  "events": [ /* envelope objects */ ]
}
```

### Batching

- Batch size MUST NOT exceed 500 events or 1 MB, whichever is smaller.
- Reporters SHOULD flush every 10 seconds or every 50 events, whichever comes first.
- Reporters SHOULD retry transient failures (5xx, network errors) with exponential backoff up to 24 hours. The `event_id` provides idempotency; the collector MUST deduplicate by `event_id`.

### Response

`202 Accepted` with a per-event result:

```json
{
  "accepted": 47,
  "rejected": 3,
  "results": [
    { "event_id": "...", "status": "accepted" },
    { "event_id": "...", "status": "rejected", "reason": "schema_invalid", "detail": "data.level out of range" }
  ]
}
```

`results` MAY be omitted when all events were accepted. Reporters MUST treat any `2xx` as durable acceptance for the events listed under `accepted` and SHOULD log rejections.

### Auth

The collector issues a long-lived bearer token to each reporter at onboarding. Tokens MAY be rotated. There's no client credentials flow in v1; the token is the credential.

### Rate limiting

Collectors MAY rate-limit per token. The standard `Retry-After` header MUST be honored by reporters on `429`.

## Identity

If [EduSSO](../edu-sso/) is in play, the `child_ref` SHOULD include the `(iss, sub)` pair from the SSO token. If only an email is available, include it. If the collector has assigned its own `child_id` for the relationship, use that and skip the rest.

Reporters MUST NOT emit events for a child they have not seen interact with their app in the current session. No backfilling identities you guessed.

---

## What v1 deliberately does *not* include

- A shared skill or topic taxonomy. `skill_id` and `topic` are app-defined.
- Streaming (WebSocket, SSE, gRPC). Batched HTTPS only.
- Subscribe / fan-out from collector to others. Collectors that want to forward do it on their own.
- Queries from reporter to collector. Reporters write; they don't read.
- Analytics, aggregations, or computed-fields-on-ingest. The collector stores or forwards; semantics belong to the dashboard, not the protocol.
- Multi-tenant scoping inside a single token. One reporter token = one app's events.
- Deletion or rectification ("the child didn't really do that, please remove"). Out of scope for v1; collectors implement their own corrections policy.

## FAQ

**Why not xAPI / Caliper?**
Both are well-engineered and both have a sociology problem: they tried to model pedagogy and produced specs so flexible that no two implementations agree on anything meaningful. EduProgress trades expressiveness for interop. You can always emit a richer payload inside the `data` field for a specific app+collector pair; v1 just doesn't standardize that part.

**What if I need an event type that isn't in the core six?**
Propose it as an RFC. In the meantime, you can put the data inside `struggle.signal` or `activity.completed` under a vendor-prefixed key (`data["ext:my-vendor:detail"]`) and the collector can store it untyped.

**What if the collector is unreachable for hours?**
Buffer locally. `event_id` makes the retry safe. The collector deduplicates.

**Can the collector reject a whole batch?**
Yes — `4xx` for malformed batches, `5xx` for collector errors. Reporters retry `5xx` and don't retry `4xx`. Per-event rejections inside a `2xx` are normal and not retried.

**Is the collector's storage in scope?**
No. Whether the collector keeps events forever, aggregates them, or fires them off to a parent's phone is the collector's business.

**Privacy?**
The events contain PII (email, child id, observed behavior). Treat them as sensitive. The collector publishes its retention and access policy out of band.

---

## Summary

Your integration with an EduProgress collector is:

1. Receive an ingest URL and a bearer token at onboarding.
2. Emit envelope-shaped events of the six core types as they happen.
3. Batch and POST to the collector, retrying transient failures.

Nothing more. If you need more, file an RFC.
